## Two layers, one idea

The library splits into a **core** and a **platform** layer.

- **Core** &mdash; tuple errors, structured concurrency, resource cleanup, dependency injection
- **Platform** &mdash; retry logic, async context, validation adapters, routing, Electron IPC, Node wrappers

Nothing asks you to stop writing normal async functions.

```ts
import { safeTry } from "yieldless/error";
import { runTaskGroup } from "yieldless/task";

const [repoError, repo] = await safeTry(loadRepository(repoId));

if (repoError) {
  return [repoError, null] as const;
}

return await runTaskGroup(async (group) => {
  const refs = group.spawn((signal) => loadRefs(repo.path, signal));
  const status = group.spawn((signal) => loadStatus(repo.path, signal));

  return {
    repo,
    refs: await refs,
    status: await status,
  };
});
```

## What it optimizes for

<div class="yl-advice-grid">
  <div class="yl-advice-card do">
    <strong>Native control flow</strong>
    <div class="yl-copy">
      Composes around <code>Promise</code>, <code>AbortController</code>,
      <code>AsyncLocalStorage</code>, and <code>Symbol.asyncDispose</code>
      instead of a separate scheduler.
    </div>
  </div>
  <div class="yl-advice-card do">
    <strong>Visible failure paths</strong>
    <div class="yl-copy">
      If something can fail, the tuple is right there. No hidden exception
      channel unless you choose to rethrow.
    </div>
  </div>
  <div class="yl-advice-card do">
    <strong>Incremental adoption</strong>
    <div class="yl-copy">
      Use one module at a time. No requirement to wrap your application in
      a framework-specific runtime.
    </div>
  </div>
</div>

## What it is not

<div class="yl-note">
  <div class="yl-copy">
    Yieldless does not ship fibers, effect interpreters, global service
    containers, or a replacement promise implementation. If a native platform
    feature already works, Yieldless wraps it instead of competing with it.
  </div>
</div>
