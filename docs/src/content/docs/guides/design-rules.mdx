---
title: Design Rules
description: The principles behind Yieldless and the tradeoffs those principles deliberately make.
---

Yieldless is small on purpose. The design rules are what keep it small.

## 1. Prefer native language features over framework runtimes

If JavaScript or Node already has a solid primitive for the job, Yieldless uses
it.

- `Promise` and `async`/`await` for sequencing
- `AbortController` and `AbortSignal` for cancellation
- `AsyncLocalStorage` for async context in Node
- `Symbol.asyncDispose` and `await using` for resource cleanup

This rule keeps the library easy to explain to engineers who did not build the
original system.

## 2. Keep failures visible

Thrown exceptions are still useful for process-level failures and framework
boundaries, but routine operational failures should stay explicit.

```ts
const [error, value] = await safeTry(readConfig());

if (error) {
  return [error, null] as const;
}
```

That shape is intentionally repetitive. It makes failure handling obvious during
code review.

## 3. Cancellation is cooperative

`runTaskGroup()`, `all()`, `race()`, and `safeRetry()` all respect
`AbortSignal`, but they cannot cancel code that ignores the signal.

If you want cancellation to mean anything, thread the signal into the actual
I/O call:

```ts
group.spawn((signal) => runCommand("git", ["fetch"], { signal }));
```

## 4. Keep adapters thin

The schema, router, IPC, and Node modules are adapters. They are not attempts to
replace the libraries they sit beside.

- `yieldless/schema` adapts `safeParse()` and `parse()` style validators
- `yieldless/router` adapts tuple handlers to JSON responses
- `yieldless/ipc` adapts Electron’s `handle()` / `invoke()` pair
- `yieldless/node` adapts filesystem calls and subprocess execution

If an adapter starts wanting its own mini-framework, it is probably the wrong
shape.

## 5. Avoid global magic

Dependency injection is just function binding. Context is explicit in the places
where context should be explicit.

Use `inject()` for stable dependencies like loggers, repositories, or feature
flags. Use `createContext()` for request-scoped data that must flow through an
async call tree without being threaded manually through every function.

## 7. Re-throw only at the boundary that needs it

`unwrap()` exists for places that genuinely expect thrown exceptions, such as a
database transaction wrapper or a framework callback that uses exceptions to
signal rollback.

Use it sparingly:

```ts
import { safeTry, unwrap } from "yieldless/error";

await transaction(async () => {
  const result = await safeTry(writeModel());
  return unwrap(result);
});
```

Do not turn `unwrap()` into the default style or you lose the whole benefit of
tuple-first flow.
