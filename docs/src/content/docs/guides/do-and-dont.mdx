---
title: Do and Don't
description: Practical conventions that keep Yieldless code readable instead of ceremonial.
---

This page is the short list of habits that keep tuple-based code clean under
real production pressure.

## Error handling

<div class="yl-advice-grid">
  <div class="yl-advice-card do">
    <strong>Do keep the tuple close to the boundary.</strong>
    <div class="yl-copy">
      Wrap I/O once, then pass domain values through the rest of the function.
    </div>
  </div>
  <div class="yl-advice-card dont">
    <strong>Don't wrap every line in <code>safeTry()</code>.</strong>
    <div class="yl-copy">
      If a function is already synchronous and pure, let it stay ordinary code.
    </div>
  </div>
</div>

```ts
const [readError, configText] = await readFileSafe("config.json");
if (readError) return [readError, null];

const [parseError, config] = safeTrySync(() => JSON.parse(configText));
if (parseError) return [parseError, null];
```

## Cancellation

<div class="yl-advice-grid">
  <div class="yl-advice-card do">
    <strong>Do pass the signal all the way into the I/O API.</strong>
    <div class="yl-copy">
      Cancellation only matters when the transport or subprocess actually sees
      the signal.
    </div>
  </div>
  <div class="yl-advice-card dont">
    <strong>Don't assume a task group can kill arbitrary CPU work.</strong>
    <div class="yl-copy">
      `AbortSignal` is cooperative. If your code ignores it, the work keeps
      running.
    </div>
  </div>
</div>

## Retries

<div class="yl-advice-grid">
  <div class="yl-advice-card do">
    <strong>Do retry transport failures and transient infrastructure noise.</strong>
    <div class="yl-copy">
      Network timeouts, brief lock contentions, and flaky subprocess startup are
      reasonable candidates.
    </div>
  </div>
  <div class="yl-advice-card dont">
    <strong>Don't retry validation failures or deterministic domain errors.</strong>
    <div class="yl-copy">
      If the request shape is bad on attempt one, it will still be bad on
      attempt three.
    </div>
  </div>
</div>

## Dependency injection

<div class="yl-advice-grid">
  <div class="yl-advice-card do">
    <strong>Do bind stable dependencies at the edge.</strong>
    <div class="yl-copy">
      Loggers, repositories, mailers, and feature flags are good candidates for
      `inject()`.
    </div>
  </div>
  <div class="yl-advice-card dont">
    <strong>Don't build a hidden service locator around it.</strong>
    <div class="yl-copy">
      If the dependencies are not obvious from the function signature, the code
      gets harder to review.
    </div>
  </div>
</div>

## Context

<div class="yl-advice-grid">
  <div class="yl-advice-card do">
    <strong>Do use async context for request-scoped metadata.</strong>
    <div class="yl-copy">
      Trace spans, request IDs, user sessions, or a transaction handle fit well.
    </div>
  </div>
  <div class="yl-advice-card dont">
    <strong>Don't use async context as your application container.</strong>
    <div class="yl-copy">
      Configuration and stable dependencies should still be explicit.
    </div>
  </div>
</div>

## Boundaries

<div class="yl-note">
  <div class="yl-copy">
    A good rule of thumb: tuples are for work you expect to fail sometimes,
    thrown exceptions are for code that truly cannot continue. If a boundary
    requires exceptions, convert at that one spot with <code>unwrap()</code> and
    move on.
  </div>
</div>
