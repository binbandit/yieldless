Platform-native by design

Yieldless gives you the parts people actually reach for in large TypeScript backends, but it keeps them in ordinary JavaScript: tuples instead of wrapper classes, AbortSignal instead of fibers, and await using instead of a proprietary scope runtime.

The shape of the library

Section titled “The shape of the library”

Yieldless is intentionally split into two layers.

  • The core layer covers tuple errors, structured concurrency, resource cleanup, and dependency injection.
  • The platform layer adds retry logic, async context, validation adapters, routing helpers, Electron IPC, and Node wrappers.

Nothing in the library asks you to stop writing normal async functions. The goal is to make common failure-heavy code paths easier to reason about without changing the runtime beneath your feet.

What Yieldless optimizes for

Section titled “What Yieldless optimizes for”
Native control flow

The library composes around Promise, AbortController, AsyncLocalStorage, and Symbol.asyncDispose instead of standing up a separate scheduler.

Visible failure paths

If something can fail, the tuple is right there in front of you. There is no hidden exception channel unless you choose to rethrow.

Incremental adoption

You can use one module at a time. There is no requirement to wrap the entire application in a framework-specific runtime.

What it is not

Section titled “What it is not”

Yieldless is not trying to replace the JavaScript runtime. It does not ship fibers, effect interpreters, global service containers, or a replacement promise implementation. If a native platform feature is already good enough, Yieldless wraps it instead of competing with it.
